## Task 1: Pipeline State Persistence Module

**Goal:** Create `blender_addon/handlers/pipeline_state.py` -- a pure-Python checkpoint read/write module with no bpy dependency.

**File:** `Tools/mcp-toolkit/blender_addon/handlers/pipeline_state.py`

### Actions

1. Implement `save_pipeline_checkpoint(checkpoint_dir: str, state: dict) -> str`
   - Writes `{checkpoint_dir}/{map_name}_checkpoint.json` with `json.dump(state, indent=2)`
   - Creates directory if not exists
   - Returns absolute path of written file

2. Implement `load_pipeline_checkpoint(checkpoint_dir: str, map_name: str) -> dict | None`
   - Returns parsed JSON dict if file exists, else `None`

3. Implement `validate_checkpoint_compatibility(checkpoint: dict, current_spec: dict) -> tuple[bool, str]`
   - Returns `(True, "")` if seed matches and location count matches
   - Returns `(False, reason_string)` if mismatch

4. Implement `get_remaining_steps(checkpoint: dict, all_steps: list[str]) -> list[str]`
   - Returns steps from `all_steps` not in `checkpoint["steps_completed"]`

5. Implement `emit_scene_hierarchy(map_name: str, location_results: list[dict]) -> dict`
   - Returns `{"map_name": str, "generated_at": ISO8601, "objects": [...]}`
   - Each object entry: `{name, type, district, world_position, world_rotation_euler, world_scale}`
   - **Note:** This function uses `bpy.data.objects` -- mark it as Blender-dependent with a guard: `if bpy is None: raise RuntimeError(...)`

Tests to write:

- `test_load_returns_none_when_no_file` -- returns None for missing file
- `test_validate_compatible_checkpoint` -- same seed + count = compatible
- `test_validate_incompatible_seed` -- different seed = (False, reason)
- `test_validate_incompatible_location_count` -- different count = (False, reason)
- `test_get_remaining_steps_all_missing` -- returns full list when checkpoint empty
- `test_get_remaining_steps_partial` -- skips already-completed steps
- `test_get_remaining_steps_all_done` -- returns empty list

**Acceptance:** All 9 tests pass. Module imports without bpy.

## Task 2: Compose_map Checkpoint Integration

**Goal:** Add `checkpoint_dir` and `resume` params to `asset_pipeline action=compose_map` in `blender_server.py`. After each step, write checkpoint. On resume=True, load checkpoint and skip completed steps.

**File:** `Tools/mcp-toolkit/src/veilbreakers_mcp/blender_server.py`

### Actions

1. Add params to `asset_pipeline()` signature:
   ```python
   checkpoint_dir: str | None = None,   # if set, checkpointing enabled
   resume: bool = False,                 # if True + checkpoint_dir set, resume from last checkpoint
   force_restart: bool = False,          # clear checkpoint and restart from scratch
   ```

2. At start of `compose_map` action block:
   - If `checkpoint_dir` and `resume=True`: call `load_pipeline_checkpoint(checkpoint_dir, map_name)`
   - If checkpoint found: validate compatibility (`validate_checkpoint_compatibility`)
   - If incompatible and not `force_restart`: return error JSON with guidance
   - If `force_restart`: delete checkpoint file and proceed fresh
   - Restore `steps_completed`, `created_objects`, `location_results`, `interior_results` from checkpoint

3. Wrap each of the 9 existing steps with checkpoint save after success:
   - After each `steps_completed.append(...)`, call `save_pipeline_checkpoint(checkpoint_dir, current_state_dict)`
   - Only save if `checkpoint_dir` is not None

4. Add `checkpoint_path` and `checkpoint_dir` to the result dict when checkpointing is active

5. Import `pipeline_state` at top of compose_map block (lazy import inside the action block to avoid circular import issues)

### Test additions to: `Tools/mcp-toolkit/tests/test_compose_planners.py`

Add to existing file:
- `test_compose_map_checkpoint_params_accepted` -- mock blender, verify no TypeError on new params
- `test_resume_skips_completed_steps_in_state` -- unit test the step-skipping logic in isolation

**Acceptance:** `asset_pipeline action=compose_map` accepts new params without breaking existing callers (no param = same behaviour as before). Checkpoint JSON file is written to `checkpoint_dir` on each step.

## Task 3: Map Package Export Action

**Goal:** Add `asset_pipeline action=generate_map_package` that takes a list of object names (from a prior compose_map result), runs game_check on all of them, generates LODs, exports one FBX per Addressable group, and writes the scene hierarchy JSON.

**File:** `Tools/mcp-toolkit/src/veilbreakers_mcp/blender_server.py`

### Actions

1. Add `"generate_map_package"` to the `asset_pipeline` action Literal type

2. Add params:
   ```python
   map_package_spec: dict | None = None,
   # map_package_spec schema:
   # {
   #   "map_name": str,
   #   "objects": [str, ...],          # from compose_map result objects_created
   #   "locations": [{...}, ...],       # from compose_map result locations
   #   "export_dir": str,               # base directory for FBX files
   #   "unity_assets_path": str | None, # defaults to ~/VeilBreakers3DCurrent/Assets/Maps/
   #   "generate_lods": bool,           # default True
   #   "skip_game_check": bool,         # default False (use only for debug)
   # }
   ```

3. Implement `generate_map_package` action block:

   **Step 1 -- Game-readiness check:**
   - For each object in `spec["objects"]`, call `blender_mesh action=game_check object_name={name}`
   - Collect failures. If any failures and not `skip_game_check`: return error with failure list.

   **Step 2 -- LOD generation:**
   - If `spec.get("generate_lods", True)`: for each mesh object, call `handle_generate_lods({"object_name": name})`
   - Collect LOD variant names (append to per-object metadata)

   **Step 3 -- Group objects by Addressable group:**
   - Call `derive_addressable_groups(map_name, locations)` from pipeline_state module
   - Assign each object to its group based on object name prefix and location type

   **Step 4 -- Export FBX per group:**
   - For each group: select all objects in that group, call `handle_export_fbx({"filepath": export_path, "selected_only": True})`
   - `export_path = {export_dir}/{map_name}/{group_name}.fbx`

   **Step 5 -- Emit scene hierarchy JSON:**
   - Call `emit_scene_hierarchy(map_name, locations)` (bpy-dependent)
   - Write JSON to `{export_dir}/{map_name}/scene_hierarchy.json`

   **Step 6 -- Derive Addressable group config:**
   - Call `derive_addressable_groups(map_name, locations)` to get group list
   - Include in result as `addressable_groups` for use by Unity `setup_map_streaming`

4. Return result dict:
   ```python
   {
       "status": "success" | "partial" | "error",
       "map_name": str,
       "fbx_files": [str, ...],          # absolute paths
       "scene_hierarchy_json": str,       # absolute path
       "addressable_groups": [{...}, ...],
       "game_check_failures": [...],
       "lod_variants_generated": int,
       "next_steps": [
           "Import FBX files into Unity Assets/Maps/{map_name}/",
           "Run: unity_world action=setup_map_streaming with scene_hierarchy_json path",
       ]
   }
   ```

### Test additions to: `Tools/mcp-toolkit/tests/test_pipeline_state.py`

- `test_derive_addressable_groups_produces_terrain_tiers` -- verifies Near/Mid/Far terrain groups
- `test_derive_addressable_groups_per_location_type` -- each location type produces one group
- `test_scene_hierarchy_fields_present` -- emitted dict has required keys per object

**Acceptance:** `asset_pipeline action=generate_map_package` runs end-to-end on a Blender scene containing at least terrain + one building, produces FBX + scene_hierarchy.json, and result contains `addressable_groups` list.

## Task 4: Unity Map Streaming Setup Action

**Goal:** Add `unity_world action=setup_map_streaming` that reads a scene hierarchy JSON and generates a C# editor script that creates Addressable groups, assigns assets, sets Static flags, and bakes occlusion.

**File:** `Tools/mcp-toolkit/src/veilbreakers_mcp/unity_server.py`
**File:** `Tools/mcp-toolkit/src/veilbreakers_mcp/shared/unity_templates/world_streaming_templates.py` (NEW)

### Actions

1. Create `world_streaming_templates.py` with two functions:

   **`generate_map_streaming_script(scene_hierarchy: dict, namespace: str = "") -> str`**
   - Reads `scene_hierarchy["objects"]` to build asset path list per Addressable group
   - Calls `generate_addressables_config_script(groups, build_remote=False)` from `build_templates.py`
   - Appends C# code that:
     - Iterates each object in scene hierarchy
     - Finds asset by path (`AssetDatabase.LoadAssetAtPath`)
     - Adds to correct Addressable group entry
     - Sets `GameObjectUtility.SetStaticEditorFlags` for building/terrain objects
     - Calls `StaticOcclusionCulling.Compute()` at end
   - Returns complete C# source string

   **`generate_occlusion_zone_spec(door: dict, room_bounds: dict) -> dict`**
   - Pure Python -- no bpy, no Unity deps
   - Takes door dict `{position, width, height, normal}` and room bounds dict
   - Returns `{portal_name, portal_vertices, zone_name, zone_bounds}` for Blender-side portal creation
   - Used by Task 3 occlusion portal geometry

2. Add `"setup_map_streaming"` to the unity_world action Literal in `unity_server.py`

3. Add param: `scene_hierarchy_json_path: str | None = None`

4. Implement action block:
   - Read `scene_hierarchy_json_path` and parse JSON
   - Call `generate_map_streaming_script(scene_hierarchy)`
   - Write to Unity project under `Assets/Editor/VeilBreakers_MapStreaming_{map_name}.cs`
   - Return `{"script_path": str, "next_steps": ["Recompile Unity", "Run VeilBreakers/World/Setup Map Streaming from menu"]}`

### Test file: `Tools/mcp-toolkit/tests/test_world_streaming_templates.py` (NEW)

Tests to write:
- `test_generate_map_streaming_script_is_valid_csharp` -- output contains `using UnityEditor`, class declaration, `Execute()` method
- `test_generate_map_streaming_includes_addressable_groups` -- each group from scene_hierarchy appears in output
- `test_generate_map_streaming_sets_static_flags` -- output contains `SetStaticEditorFlags`
- `test_generate_map_streaming_calls_occlusion_compute` -- output contains `StaticOcclusionCulling.Compute`
- `test_generate_occlusion_zone_spec_returns_required_fields` -- returns portal_name, portal_vertices, zone_name, zone_bounds

## Task 5: PIPE-01 AAA Techniques Research Document

**Goal:** Write `.planning/research/PIPE-01-AAA-TECHNIQUES.md` documenting AAA procedural techniques with implementation notes for each technique used or planned in the VeilBreakers pipeline.

**File:** `.planning/research/PIPE-01-AAA-TECHNIQUES.md`

### Actions

1. Research and document each required technique with:
   - **What it is:** definition and purpose
   - **AAA game reference:** which game uses it and how (Skyrim, Fable, AC Valhalla, or equivalent)
   - **Current implementation status:** what exists in this codebase today
   - **VeilBreakers adaptation:** how it is or should be implemented

2. Required sections:
   - **CGA Split Grammars** -- building facade generation (Phase 32 building_quality.py reference)
   - **Wave Function Collapse (WFC)** -- tile-based layout generation
   - **L-systems** -- road networks, vegetation (road_network.py + vegetation_lsystem.py reference)
   - **Hydraulic Erosion** -- terrain realism (50K+ droplets) (_terrain_erosion.py reference)
   - **Poisson Disk Sampling** -- prop/vegetation scatter (prop_density.py reference)
   - **Straight Skeleton Roofs** -- roof geometry from building footprints
   - **Domain Warping** -- noise-on-noise for organic biome boundaries (map_composer.py biome noise)

3. For each technique, assess implementation status:
   - IMPLEMENTED -- code path identified, cite file + function
   - PARTIAL -- exists but not wired into compose_map pipeline
   - PLANNED -- not yet implemented, note which phase owns it
   - DEFERRED -- out of scope for v7.0

### Acceptance

Document exists at `.planning/research/PIPE-01-AAA-TECHNIQUES.md`. Every required technique from the PIPE-01 requirement has a section. At least 3 techniques have concrete file/function citations from this codebase. Document is 300+ words.
