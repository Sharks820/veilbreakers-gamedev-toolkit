    2. handle_preview_animation(params) -> dict (ANIM-07):
       Params: object_name, action_name (optional, uses active action), frame_step (default 4), angles (default ["front", "side"]), output_path (optional), resolution (default 256).
       - Set the armature's active action to the specified action
       - Calculate total frames from action.frame_range
       - For each frame (stepping by frame_step), set scene.frame_set(frame) then render from each angle
       - REUSE existing handle_render_contact_sheet by calling it with appropriate params:

       - ALTERNATIVE if contact sheet doesn't support frame-stepping natively: iterate frames manually, render each via _render_viewport_to_file (from viewport.py internals), then compose_contact_sheet
       - Return {"image_path", "frame_count", "frame_step", "angles", "cells"}

    3. handle_add_secondary_motion(params) -> dict (ANIM-08):
       Params: object_name, action_name (required), bone_names (list of spring bone names), frame_start (optional), frame_end (optional).
       - This handler BAKES existing spring bone constraint simulations to keyframes
       - Set the action as active, ensure spring bone constraints exist on specified bones (from Phase 4 RIG-06)
       - Use bpy.ops.nla.bake() with:
         frame_start, frame_end from action.frame_range
         only_selected=False (or select spring bones first)
         bake_types={'POSE'}
         visual_keying=True (captures constraint effects)
       - After baking, the spring bone channels have explicit keyframes
       - Return {"action_name", "baked_bones": list, "frame_range": [start, end]}

    4. handle_extract_root_motion(params) -> dict (ANIM-09):
       Params: object_name, action_name (required), hip_bone (default "DEF-spine"), root_bone (default "root"), extract_rotation (default False).
       - Set action as active on armature
       - Frame-by-frame: read hip bone's world-space XY translation
       - Create/get root bone fcurves for location X, Y (and rotation Z if extract_rotation)
       - Insert hip XY values as root bone keyframes
       - Subtract extracted XY from hip bone's local fcurves (so hip stays in-place)
       - Keep hip Z (vertical bob) intact
       - Add animation events: pose_markers at contact frames (foot_contact_0, foot_contact_1, etc.) extracted from existing markers or computed from foot bone Z position minima
       - Return {"action_name", "root_bone", "frames_processed", "total_distance", "events_placed"}

       PITFALL AVOIDANCE (from RESEARCH):
       - Process frame-by-frame, not by modifying raw fcurve values
       - Keep hip Z intact to preserve vertical bob
       - Check pose_bone.rotation_mode before accessing rotation channels

    Write tests: TestPreviewParams (frame_step validation, angles), TestSecondaryMotionParams (bone_names required, frame range), TestRootMotionParams (action_name required, hip_bone default).

    1. MIXAMO_TO_RIGIFY dict (module-level constant):
       Full mapping from Mixamo bone names to Rigify DEF bones. At minimum:
       - Spine chain: Hips, Spine, Spine1, Spine2, Neck, Head (6 bones)
       - Left arm: LeftShoulder, LeftArm, LeftForeArm, LeftHand (4 bones)
       - Right arm: RightShoulder, RightArm, RightForeArm, RightHand (4 bones)
       - Left leg: LeftUpLeg, LeftLeg, LeftFoot, LeftToeBase (4 bones)

       Total: 22 core mappings. All keys prefixed with "mixamorig:".

    2. handle_retarget_mixamo(params) -> dict (ANIM-10):
       Params: object_name (target Rigify rig), source_file (FBX path with Mixamo animation), action_name (optional, name for new action).
       - Import the Mixamo FBX: bpy.ops.import_scene.fbx(filepath=source_file)
       - Find the imported armature (has "mixamorig:" bones)
       - Build bone mapping using MIXAMO_TO_RIGIFY, filtered to bones that exist on both rigs
       - Use constraint-based transfer approach (matching Phase 4 retarget pattern):
         For each mapped pair, add Copy Rotation constraint from source to target
       - Bake the constrained animation: bpy.ops.nla.bake() with visual_keying=True
       - Remove constraints and imported source armature
       - Return {"action_name", "mapped_bones": int, "unmapped_bones": list, "frame_range"}

    3. handle_generate_ai_motion(params) -> dict (ANIM-11):
       Params: object_name, prompt (text description), model (default "hy-motion"), frame_count (default 48).
       - This is a STUB -- HY-Motion has no HTTP API yet
       - Validate params
       - Return {"status": "stub", "message": "AI motion generation not yet available. HY-Motion/MotionGPT integration pending API availability.", "prompt": prompt, "model": model, "frame_count": frame_count}
       - When API becomes available, this handler will:
         1. Call AI model with text prompt
         2. Receive skeleton joint rotations (BVH or raw)
         3. Convert to Blender Action
         4. Retarget to custom rig

    4. handle_batch_export(params) -> dict (ANIM-12):
       Params: object_name, output_dir (required), naming (default "unity"), actions (optional list of action names to export, default all).
       - Validate output_dir exists (create if not)
       - Push each action to NLA as separate strip using _push_action_to_nla helper
       - For each NLA strip:
         a. Solo the strip (mute all others)
         b. Select only the armature + its mesh children
         c. Export FBX with: bake_anim=True, bake_anim_use_nla_strips=True, bake_anim_use_all_actions=False, bake_anim_force_startend_keying=True, apply_scale_options="FBX_SCALE_ALL", axis_forward="-Z", axis_up="Y", add_leaf_bones=False
         d. Unity naming: "{object_name}@{strip_name}.fbx"
       - Restore all strip mute states after export
       - Return {"exported_clips": count, "clips": [{"name", "filepath", "frame_range"}], "output_dir"}

       Helper: _push_action_to_nla(armature_obj, action, strip_name=None)
       Helper: _solo_nla_strip(anim_data, target_strip) -- mutes all strips except target
       Helper: _restore_nla_mute_states(anim_data, saved_states) -- restores original mute states

       PITFALL AVOIDANCE (from RESEARCH):
       - Set BOTH bake_anim_use_nla_strips=True AND bake_anim_use_all_actions=False
       - Without both flags, Unity gets one "Take 001" instead of individual clips

    Write tests: TestMixamoBoneMapping (completeness, specific mappings, all have DEF prefix), TestAIMotionStub (returns stub status, validates params), TestBatchExportParams (output_dir required, naming validation, Unity naming format).
